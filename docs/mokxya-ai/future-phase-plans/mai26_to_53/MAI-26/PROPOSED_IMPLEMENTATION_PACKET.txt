1. PHASE OBJECTIVE (PROPOSED)
----------------------------------------------------------------------

Deliver a governed temporal resolution layer that:
  1. Models valid time and system time for document versions and knowledge units.
  2. Records amendment, supersession, repeal, and commencement relationships
     without overwriting prior versions.
  3. Stores structured rates/thresholds with explicit valid intervals and unit
     provenance.
  4. Builds a cross-reference graph between citation-ready units.
  5. Resolves as-of queries for old and current provisions, failing closed on
     conflicting or unknown effective dates.
  6. Keeps historical versions accessible after supersession/repeal.

WS-A — Domain model and schemas

    SystemIntervalV1 (recorded_from, recorded_to / release_id)

      valid interval, source_unit_id, content_hash)
    TemporalQueryV1 / TemporalResolutionV1
  Persist relationship and rate tables per roadmap knowledge table sketch,
  reconciled to MAI-24/25 final schema names (UNRESOLVED_PENDING_HANDOFF).

WS-B — Ingestion linking service
  After MAI-25 segmentation, add deterministic linker stage:
    extract explicit cross-ref mentions where structured;
    attach amendment/supersession edges only with reviewed or high-confidence
      structured metadata;
    never mutate raw artifacts or prior DocumentVersion rows;
    queue ambiguous links for human review (MAI-24 review workflow).

WS-C — Temporal resolver application service
  Proposed service: temporal_resolver (roadmap legal_runtime/application)
  Inputs: tenant/jurisdiction, instrument or unit hint, as_of_valid_time,
    optional as_of_system_time / release_id
  Outputs:
    RESOLVED (one or ordered set with relationship trail)
    CONFLICT (multiple incompatible versions; no silent pick)
    UNKNOWN_EFFECTIVE (missing/partial dates; block definitive)
    NOT_IN_FORCE / REPEALED (with successor pointers if known)
  Must support:
    FY/historical queries without destroying current release views
    "what did repository believe on date X" via system time / release

WS-D — Integration with existing retrieval pipeline (careful overlap)
  Extend or replace TemporalFilterStage semantics so historical queries can
  retrieve superseded versions when as_of demands it.
  Authority/time filters must run before final evidence (feeds MAI-29/30).
  Do not implement full hybrid retrieval (MAI-27–29) inside MAI-26.
