1. WHAT WE ARE BUILDING (REMINDER)
======================================================================

Public product: Ask MokXya inside MokXya Hisab / MokXya ERP.

Target (from master §1 — still correct):

  ChatGPT-like conversation quality inside a controlled Nepal accounting,
  financial-reporting, legal, tax, and business domain — and *better than*
  a general assistant for:
    - company-scoped answers
    - informal shopkeeper language → structured accounting previews
    - deterministic calculations and safe posting
    - evidence-based Nepal law/tax research (with uncertainty)

Not the target:

  Universal ChatGPT clone
  Autonomous posting from model confidence
  Guaranteed legal conclusions without reviewed sources
  “Fully automatic CA / lawyer” marketing

Core UX that must work end-to-end for launch:

  User types/speaks EN / Nepali Devanagari / Romanized / mix
  → MokXya understands in context
  → retrieves authorized facts
  → answers, clarifies, or prepares a typed preview
  → user reviews/edits
  → nothing posts until explicit scoped confirmation
  → deterministic services validate and execute
  → real receipt + honest sync state

2. HOW ADVANCED IT SHOULD BE (HONEST BAR)
======================================================================

“Advanced” for MokXya means these bars, in order:

BAR A — SAFE
  Wrong mutation, cross-tenant leak, or false legal current-law claim is blocked.
  Ask Mode cannot mutate. NL “yes/huncha” alone cannot post.

BAR B — USEFUL IN SHOP LANGUAGE
  English + Devanagari + Romanized + code-mix produce stable slots for supported
  events. Number/date/money roles do not treat “5 months” as NPR 5.
  New topics do not silently merge into stale drafts.

BAR C — USEFUL FOR BOOKKEEPING
  Supported purchase/sales/payment-class events go draft → preview → confirm →
  receipt on one documented write authority, with idempotency and audit.

BAR D — HONEST KNOWLEDGE
  No-answer when evidence is missing. Citations entail claims. Legal/tax
  currentness is never invented. Prod retrieval does not depend on a local-only
  Ollama/Chroma stack the cloud never runs.

BAR E — NARROW RELEASE
  One capability matrix row can be PRODUCTION_APPROVED with monitoring,
  rollback, and disclosed limitations — not “all of AI is live.”

BAR F — PROFESSIONAL EXTENSIONS (LATER)
  Speech, private documents, CA workpapers, compliance calendars — only after
  Bars A–E for the launch slice. Track I in the master file remains future work;
  current MAI-50…53 engineering stubs are *not* those capabilities.
