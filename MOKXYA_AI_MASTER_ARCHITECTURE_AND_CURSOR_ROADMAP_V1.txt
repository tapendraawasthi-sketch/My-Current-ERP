0. PURPOSE OF THIS DOCUMENT
======================================================================

This document defines the architecture MokXya must gradually develop if it is to
feel close to a high-quality general AI assistant within a controlled Nepal
accounting, financial-reporting, legal, tax, and business domain.

It has four purposes:

1. Preserve the product vision and all non-negotiable safety boundaries.
2. Define the target code architecture, data contracts, knowledge architecture,
   language-learning program, evaluation system, and production controls.
3. Give future ChatGPT conversations a stable source from which to generate one
   exact Cursor implementation prompt at a time.
4. Prevent Cursor, or any other coding AI, from creating another disconnected
   prototype beside systems already present in the repository.

This file is not a single “implement everything” Cursor prompt. It is the master
specification from which small, auditable, test-driven prompts must be produced.

This file complements the existing repository-specific document:

  MOKXYA_HISAB_MASTERPIECE_AI_LAUNCH_DIRECTION_V1.txt

If the two documents differ because the repository changed, Cursor must first
inspect the current code and produce evidence. Safety, accounting authority,
tenant isolation, explicit confirmation, truthful evidence, and deterministic
posting rules always take precedence.

HOW TO USE THIS FILE IN FUTURE CHATGPT CONVERSATIONS

1. Upload or reference this file together with the latest repository or latest
   Cursor phase-completion report.
2. Tell ChatGPT which phase was last proven complete.
3. ChatGPT must inspect the evidence and select only the earliest safe incomplete
   phase.
4. ChatGPT then generates one repository-aware Cursor prompt for that phase.
5. Cursor must inspect before editing, add failing tests first, implement the
   smallest coherent change, run all specified gates, and return evidence.
6. ChatGPT reviews the evidence. It must not accept “done” merely because Cursor
   says so or because compilation succeeds.
7. Only after the gate is proven should the next phase be generated.

1. EXECUTIVE VERDICT: WHAT IS AND IS NOT ACHIEVABLE
======================================================================

MokXya cannot reproduce ChatGPT’s universal capability by collecting vocabulary
files, writing a long system prompt, using RAG alone, or placing a 32B/37B model
behind a chat box. General ChatGPT-level breadth requires enormous pretraining,
post-training, infrastructure, evaluation, and continuous research.

The realistic and valuable target is different:

  MokXya can become ChatGPT-like in conversation quality and better than a
  general assistant in supported Nepal accounting workflows, company-specific
  answers, Nepal legal research, evidence tracking, deterministic calculations,
  and safe ERP execution.

This requires a composed intelligence system:

  capable multilingual foundation model
  + Nepali/Romanized language runtime
  + structured conversation state and memory
  + intent/event understanding
  + planner and typed tool use
  + verified company-data retrieval
  + authoritative legal/business RAG
  + deterministic accounting/tax/reporting engines
  + answer and action verification
  + security, audit, observability, and evaluation

EXPECTED ACHIEVABLE OUTCOMES

1. Natural English, Devanagari Nepali, Romanized Nepali, and mixed-language
   conversation for the supported domain.
2. Correct resolution of follow-up messages, corrections, pronouns, omitted
   subjects, and references to active drafts in controlled workflows.
3. High-quality answers grounded in authorized company data or reviewed sources.
4. Correct structured accounting previews from informal shopkeeper language.
5. Safer and more reliable entries than a general chat model because calculations
   and posting are deterministic.
6. Evidence-based Nepal law and tax research with version, effective date,
   authority, citations, and uncertainty.
7. Automatic financial-reporting workflows where rules and mappings are explicitly
   implemented and professionally validated.

NOT HONESTLY ACHIEVABLE

1. Perfect understanding of every dialect, spelling, or unseen expression.
2. Guaranteed correct legal/tax conclusions without authoritative current sources
   and qualified human review.
3. An AI that safely invents missing accounting facts.
4. Autonomous posting merely because the model is confident.
5. Permanent legal freshness without a governed update and review process.
6. ChatGPT-level universal world knowledge from a small domain dataset.
7. Reliable production quality from prompt engineering without tests and metrics.

3. NON-NEGOTIABLE PRODUCT CONSTITUTION
======================================================================

These constraints must be enforced in domain code, APIs, database rules,
authorization, and tests. A prompt is not an adequate enforcement boundary.

3.1 LANGUAGE MODEL AUTHORITY

The model may:
  understand language;
  classify intent and dialogue acts;
  propose structured interpretations;
  identify ambiguity and missing information;
  plan permitted reads and tools;
  summarize retrieved evidence;
  draft explanations;
  propose a typed action draft.

The model may not:
  become the accounting ledger authority;
  calculate authoritative tax or journal results without deterministic validation;
  invent a party, item, amount, date, account, tax treatment, document, citation,
  legal provision, rate, threshold, or company fact;
  directly write to ERP databases;
  bypass permissions, preview, confirmation, validation, or OEC;
  claim an action succeeded without a connector receipt;
  expose hidden reasoning or provider think blocks.

3.2 ASK MODE

Ask Mode is read-only. It can explain, search, calculate through approved read-only
tools, query authorized reports, and show clearly labeled simulations. It cannot
create or confirm a mutation, even if the user uses imperative language.

3.3 ACCOUNTANT MODE

Accountant Mode can prepare supported typed drafts. Every mutation requires:

1. authenticated tenant/company/user context;
2. permission and capability checks;
3. a versioned structured draft;
4. deterministic validation and preview;
5. an explicit short-lived, single-use confirmation token;
6. OEC execution with idempotency;
7. immutable audit and lineage;
8. a real receipt and honest sync state.

A natural-language “yes”, “ok”, “huncha”, or “thik cha” must not alone serve as
posting authority.

3.4 ACCOUNTING AUTHORITY

Do not alter established posting commands, VAT/TDS calculations, inventory
valuation, period locks, permissions, voucher numbering, sync semantics, or audit
rules from a language/UI phase. The UI and LLM do not calculate authoritative facts
or create parallel posting paths.

3.5 TRUTH AND EVIDENCE

Every material answer must identify its evidence class:

  live ERP fact;
  deterministic calculation;
  reviewed external authority;
  product documentation;
  general explanation;
  unresolved interpretation.

Current legal, tax, regulatory, or reporting claims require:

  jurisdiction;
  authoritative source;
  document/version identity;
  effective date or period;
  as-of date;
  precise cited provision where available;
  review status;
  conflict or uncertainty disclosure.

3.6 TENANT AND PRIVACY ISOLATION

Tenant, company, fiscal year, branch, user, role, and permissions come from trusted
authenticated context. They must scope every message, memory, cache, retrieval,
draft, preview, command, event, audit record, trace, and metric. Development default
identities are forbidden in production paths.

3.7 SAFE FAILURE

When a dependency is unavailable, MokXya must degrade honestly:

  no ERP evidence -> no claim about live company facts;
  no current reviewed law -> no definitive current-law answer;
  no posting service -> no fabricated voucher;
  ambiguous event -> focused clarification;
  unsupported event -> safe explanation/escalation;
  model outage -> explicitly limited deterministic/read-only functions;
  failed import/migration/security gate -> production readiness fails.

3.8 RESPONSE LANGUAGE

Default to the language and register of the user’s latest meaningful message while
preserving technical clarity. A mixed Romanized Nepali query may receive natural
Romanized Nepali with important official terms also shown in Nepali or English when
useful. Never mechanically translate stable names, account IDs, invoice numbers,
citations, code, PAN/VAT numbers, or legal references.

4. ARCHITECTURAL PRINCIPLES
======================================================================

1. One authoritative intelligence pipeline, not several chat systems competing.
2. Modular monolith first unless measured scaling requires service separation.
3. Domain boundaries and typed ports before provider-specific integrations.
4. Structured contracts between probabilistic and deterministic components.
5. Original user text is immutable; normalization produces annotations and views.
6. Language form is detected before intent classification.
7. Script is not the same as language.
8. Evidence spans accompany extracted values.
9. Ambiguity is represented, not silently removed.
10. Model confidence is not accounting or legal confidence.
11. Retrieval is not proof; a verifier must connect claims to evidence.
12. Law, rates, and accounting policies are time/version dependent.
13. Every derived index is rebuildable from immutable validated sources.
14. Every mutation is previewed, explicitly confirmed, idempotent, and audited.
15. No switch-statement jungle for providers, tools, intents, or connectors; use
    registries and declared capabilities.
16. No global mutable conversation state.
17. No raw chain-of-thought storage or exposure.
18. No fine-tuning until the architecture and baseline evaluations are trustworthy.
19. Every phase is reversible and has an acceptance gate.
20. Optimize measured bottlenecks; do not trade away safety for superficial speed.

5. TARGET END-TO-END INTELLIGENCE PIPELINE
======================================================================

CANONICAL READ/ANSWER FLOW

  API Gateway
    -> Authentication and Tenant Context
    -> Request Validation and Rate/Size Control
    -> Conversation State Load
    -> Language Form and Protected-Span Analysis
    -> Lossless Normalization and Candidate Transliteration
    -> Turn-Relation and Dialogue-Act Classification
    -> Intent/Event Candidate Generation
    -> Context and Reference Resolution
    -> Capability/Mode Policy
    -> Planner
    -> Typed ERP/Knowledge/Calculation Tool Reads
    -> Evidence Bundle
    -> Structured Interpretation or Answer Plan
    -> Model Composition
    -> Claim, Citation, Language, and Safety Verification
    -> Typed Response Envelope
    -> SSE/UI Rendering
    -> Redacted Trace and Evaluation Hooks

CANONICAL ACTION FLOW

  User Turn
    -> same understanding pipeline
    -> EventFrame with evidence and uncertainties
    -> Required-Field and Ambiguity Policy
    -> Focused Clarification OR Draft Builder
    -> Deterministic Domain Validation
    -> Versioned Preview
    -> User Review/Edit
    -> Explicit Confirmation Token
    -> Permission/Snapshot/Revalidation
    -> Action Runtime
    -> ERPCommandPort
    -> OEC
    -> Connector
    -> ERP Authority
    -> Receipt, Audit, Outbox, Sync State
    -> User-visible Result

The LLM never jumps from raw text directly to the connector.

7. LANGUAGE INTELLIGENCE ARCHITECTURE
======================================================================

MokXya does not need to manually store every possible word. It needs a strong
multilingual base model plus a controlled language runtime, domain ontology,
training examples, retrieval, and evaluation.

7.1 REQUIRED LANGUAGE FORMS

Support and label:

1. Formal Nepali in Devanagari.
2. Informal conversational Nepali in Devanagari.
3. Formal English.
4. Nepal-business English.
5. Romanized Nepali with common variants.
6. English-Nepali code-mix.
7. Devanagari-Romanized code-mix.
8. Three-way English-Devanagari-Romanized input.
9. Abbreviations and accounting shorthand.
10. Typographical errors, missing spaces, repeated characters, and mobile typing.
11. Nepali and English numerals.
12. BS and AD dates and fiscal-year expressions.
13. Locally common Hindi-influenced forms only where observed, without expanding
    product scope to a separate Hindi assistant.

7.2 SCRIPT AND LANGUAGE-FORM DETECTION

Detection operates at token/span level, not only whole-message level. It must
identify:

  Unicode script;
  likely language form;
  Romanized Nepali likelihood;
  English likelihood;
  named entity;
  identifier/code;
  number/date/money;
  URL/email/phone;
  legal citation;
  account or invoice reference;
  protected text that must not be transliterated.

Examples:

  “Ram lai 50000 udharo sales गरियो”

must not be classified as merely English or merely Nepali. “Ram” may be a party
name, “50000” an amount candidate, “udharo” Romanized Nepali, “sales” English, and
“गरियो” Devanagari Nepali.

7.3 LOSSLESS NORMALIZATION

Normalization is layered and reversible:

  Layer 0: preserve original bytes/text.
  Layer 1: safe Unicode normalization.
  Layer 2: whitespace and punctuation annotation.
  Layer 3: spelling/abbreviation candidate expansion.
  Layer 4: Romanized Nepali canonical candidates.
  Layer 5: domain synonym mapping.
  Layer 6: entity/master matching candidates.

Never silently replace a person, item, account, number, invoice, PAN/VAT number,
legal section, or user-defined label.

Every edit records:
  original span;
  candidate;
  rule/model version;
  confidence;
  whether safe for retrieval only or safe for semantic interpretation.

7.4 ROMANIZED NEPALI

Romanized Nepali has no single spelling standard. The language runtime must accept
many-to-many mappings such as:

  cha / chha / xa -> छ
  bhayo / vayo / vo -> भयो
  udharo / udaro / udhaar -> उधारो
  kine / kinyo / kineko -> खरिद-related candidate depending on context
  beche / bechye / bikri gare -> sale-related candidate depending on context

Do not force one transliteration before understanding. Generate n-best candidates
and use context, domain ontology, company masters, and the model to rank them.

Maintain:
  phoneme/grapheme mappings;
  common spelling variants;
  morphology-aware suffix handling;
  word-boundary variants;
  common mobile-keyboard errors;
  regional/informal variants with provenance;
  protected-entity policy;
  bidirectional transliteration evaluation.

7.5 CODE-MIX HANDLING

Code-mixing is normal, not an error. The system must preserve technical English
terms when users naturally use them:

  “Kiran ko ledger dekhaau”
  “yo invoice ko VAT milaena”
  “cash ma purchase gareko ho”
  “गत महिनाको sales report चाहियो”

Intent and slot models must be trained and tested on mixed spans, not only fully
translated sentences.

7.6 NUMBERS, MONEY, QUANTITY, DURATION, AND IDENTIFIERS

A number is not automatically money. Required number roles include:

  amount;
  quantity;
  unit price;
  rate;
  percentage;
  discount;
  tax;
  duration;
  date part;
  fiscal year;
  invoice/reference number;
  account/PAN/VAT/phone identifier;
  installment count;
  age/tenure;
  unknown.

Example:

  “ghar bhada tirnu xa 5 maina ko”

means duration = five months and amount = unknown. It must never become Rs 5,000.

Number understanding must support:
  50000, 50,000, ५००००;
  50k where product policy allows;
  pachas hajar, पचास हजार;
  lakh/crore and लाख/करोड;
  decimals and percentages;
  Nepali currency expressions;
  relative dates and periods;
  BS/AD conversion through a verified deterministic date service.

7.7 DOMAIN LEXICON AND ONTOLOGY

The lexicon is supporting data, not the entire intelligence system. It must link
surface forms to stable concepts:

  surface form -> lemma/concept -> domain -> possible roles -> examples
  -> ambiguity notes -> language/register -> source -> version

Domains initially include:
  sales, purchase, return, receipt, payment, journal, contra, cash, bank, credit,
  discount, bad debt, provision, theft/loss, expense, income, inventory, VAT, TDS,
  payroll, fixed assets, depreciation, reconciliation, reporting, company setup.

Future domains include:
  Income Tax, VAT, Excise, Customs, Companies, Labour, Social Security, NFRS/NAS,
  auditing, NRB, banking, insurance, securities, cooperatives, insolvency, contracts,
  consumer protection, local taxes, licensing, sector regulation, court decisions,
  and administrative procedures.

7.8 RESPONSE GENERATION

The model should compose natural language from a verified answer plan. Response
quality rules include:

1. Match user language and form.
2. Be concise for simple actions and sufficiently detailed for research.
3. Use familiar language first, technical term second where helpful.
4. Preserve official titles and citations accurately.
5. Do not translate identifiers.
6. Ask one high-information-gain clarification at a time when possible.
7. State what was understood before asking when ambiguity is material.
8. Do not repeat already known facts.
9. Distinguish “I need information” from “the event is unsupported.”
10. Never expose hidden reasoning.

8. LANGUAGE DATA AND LEARNING PROGRAM
======================================================================

8.1 DATA TYPES REQUIRED

1. Raw consented/anonymized Nepal business utterances.
2. English, Devanagari Nepali, Romanized Nepali, and mixed paraphrases.
3. Multi-turn conversations rather than isolated sentences only.
4. Corrections, cancellations, topic changes, references, and incomplete replies.
5. Domain intent and event examples.
6. Entity/slot annotations with exact evidence spans.
7. Number-role annotations.
8. Ambiguity and required-clarification annotations.
9. Safe answer and unsafe-answer contrasts.
10. Tool-selection and typed-argument examples.
11. Knowledge question/evidence/citation examples.
12. Out-of-domain, adversarial, and no-answer examples.
13. Voice transcription errors and OCR noise when those channels launch.
14. Realistic dialect/register coverage with documented provenance.

8.2 ANNOTATION SCHEMA

Every critical conversation should label:

  language form by span;
  dialogue act;
  turn relation;
  primary and alternative intent;
  event lifecycle;
  entities and source spans;
  number roles;
  explicit versus inferred values;
  reference/coreference links;
  required company-data reads;
  required knowledge reads;
  missing and ambiguous fields;
  expected clarification;
  allowed capability;
  forbidden assumptions;
  expected structured result;
  expected user-facing answer style.

Critical accounting/legal cases require double annotation and adjudication. Record
reviewer role, annotation-guideline version, and disagreement.

8.3 DATA QUALITY

Reject or quarantine:
  duplicate templates spread across train and test;
  fabricated legal citations presented as truth;
  examples containing real sensitive identifiers without consent and sanitization;
  inconsistent intent labels;
  answers that assume missing facts;
  synthetically generated examples without review;
  obsolete law/rates mixed with current truth without effective dates;
  translations that erase natural Romanized/code-mixed forms.

8.4 TRAIN/VALIDATION/TEST SPLITS

Split by underlying scenario and conversation, not by individual sentence. All
paraphrases and transliteration variants of one seed remain in the same split.

Maintain:
  development training set;
  validation set;
  frozen gold test;
  adversarial safety set;
  current-law temporal set;
  post-launch shadow set;
  regional/register challenge set.

The frozen test set must not be fed into prompt development or fine-tuning.

8.5 SYNTHETIC DATA

Synthetic data is useful for coverage expansion, but must be generated from a
controlled scenario matrix, validated against schemas, deduplicated, sampled for
human review, and kept traceable to its seed and generator version.

Generate variations across:
  language form;
  spelling;
  code mix;
  word order;
  politeness/register;
  omitted fields;
  correction patterns;
  ambiguous numbers;
  multiple parties/items;
  date expressions;
  dialect and typing noise.

Do not let synthetic data define legal truth or accounting rules.

8.6 FINE-TUNING POLICY

Do not fine-tune first. Establish a strong evaluated baseline after routing,
context, tools, retrieval, and verifiers work correctly.

Use supervised fine-tuning or LoRA later for:
  language/style adaptation;
  Romanized Nepali robustness;
  structured extraction;
  dialogue-act/turn-relation classification;
  tool selection;
  clarification behavior;
  consistent schema output.

Do not use model weights as the primary store for:
  current laws;
  changing tax rates and deadlines;
  company balances;
  permissions;
  chart-of-account mappings;
  posting logic;
  effective-date rules.

Preference training may improve answer helpfulness and safe clarification only after
high-quality preference pairs and an evaluation gate exist.

8.7 MODEL AND TOKENIZER EVALUATION

Benchmark every candidate base model on:
  Nepali script token efficiency;
  Romanized Nepali understanding;
  code-mix;
  structured JSON reliability;
  long-context faithfulness;
  tool use;
  hallucination/no-answer behavior;
  accounting reasoning;
  deployment memory and latency;
  license and commercial-use conditions.

A larger parameter count does not automatically mean better Nepali performance.

9. CONVERSATION CONTEXT AND MEMORY
======================================================================

Most “weak context” failures are application-state failures, not merely model-size
failures.

9.1 CONTEXT TYPES

Maintain separate stores for:

1. Current turn: immutable raw input and analysis.
2. Working context: facts needed for the present answer.
3. Conversation context: recent turns and typed references.
4. Task/draft state: fields, versions, status, outstanding clarification.
5. Company context: authorized company, fiscal year, branch, configuration.
6. Business memory: approved durable preferences and mappings.
7. User preference memory: language/style, where allowed.
8. Knowledge evidence: retrieved sources for the current answer.
9. Audit history: immutable action/decision trace.
10. Compressed summaries: derived, versioned, replaceable aids.

Do not merge these into a single unstructured chat-history string.

9.2 CONTEXT ASSEMBLY

Build a ContextBundle with explicit budgets and priority:

  system constitution;
  capability/mode policy;
  authenticated scope;
  active task/draft;
  unresolved clarification;
  relevant recent turns;
  resolved entity references;
  authorized company facts;
  retrieved external evidence;
  relevant durable preferences;
  response contract.

Every included item records provenance and freshness. Exclude unrelated ledger data,
old drafts, and unnecessary personal/business details.

9.3 TURN RELATION BEFORE DRAFT MERGE

Never treat every message after a clarification as an answer to that clarification.
Classify whether it:

  answers the question;
  corrects a prior field;
  starts a new event;
  refers to another draft;
  cancels;
  asks an explanation;
  confirms intent but not authority;
  is unrelated.

A stale purchase draft must never capture a new theft, sale, greeting, or legal
question.

9.4 REFERENCE AND COREFERENCE

Resolve expressions such as:
  “tesko”, “tyo”, “usle”, “paila ko”, “yo entry”, “same party”, “tesmai 500 thap”,
  “last month ko”, “agiko invoice”, “that one”, “make it 450”.

Resolution candidates come from typed recent objects, not only text similarity.
If two candidates remain materially plausible, present choices.

9.5 CORRECTION SEMANTICS

Corrections create a new draft version and preserve history:

  original value;
  corrected value;
  source turn;
  user who corrected;
  timestamp;
  reason/field;
  resulting preview hash.

Previously issued confirmation tokens become invalid after any material edit.

9.6 SUMMARIZATION

Summaries are derived aids, not authority. They must:
  preserve unresolved decisions;
  preserve stable object IDs;
  distinguish user statements from tool facts;
  exclude chain of thought;
  be regenerated when source turns change;
  be invalidated when a correction contradicts them;
  never replace financial records or citations.

9.7 MEMORY WRITE POLICY

Do not automatically turn every conversation statement into durable memory.
Memory writes require:
  allowed memory type;
  relevance and expected reuse;
  tenant/user scope;
  sensitivity classification;
  retention/expiry;
  provenance;
  correction/deletion support.

Company facts should normally be queried from authoritative ERP records rather than
memorized in model conversation memory.

10.3 PLANNER

The planner produces typed, bounded plans. It does not write free-form steps that
arbitrary code executes.

Planner policies:
  prefer direct deterministic handlers for simple requests;
  retrieve only required facts;
  parallelize independent read-only tools when safe;
  limit steps and retries;
  stop when required evidence is missing;
  never convert Ask Mode into a mutation plan;
  never call an undeclared tool;
  validate every tool argument;
  prevent recursive unbounded agent loops;
  record plan and tool versions.

10.4 MULTI-AGENT USE

Do not create “agents” merely as marketing labels. Use specialized workers only
where decomposition and verification provide measured value, for example:

  research source collector;
  legal temporal resolver;
  accounting explanation composer;
  claim verifier;
  document ingestion/OCR worker.

All workers operate under one orchestrator, shared typed contracts, capability
limits, evidence lineage, time/step budgets, and final verification. No worker may
mutate ERP except through the same authorized action path.

11.2 MODEL CASCADE

The previously chosen Qwen fast model and larger reasoning model can remain a
starting cascade, but selection must be benchmark-driven.

Suggested logical roles:

  deterministic/no-model path:
    exact commands, simple navigation, known report retrieval, validation

  fast model:
    language classification, rewriting candidates, simple answers, routing,
    low-risk schema extraction

  capable model:
    ambiguous multi-turn interpretation, research synthesis, complex explanation,
    difficult structured extraction

  embedding model:
    multilingual semantic retrieval

  reranker:
    multilingual query-document relevance

  optional specialist classifier:
    small fine-tuned model for language form, turn relation, or intent if it
    materially beats the generative model on cost/latency/accuracy

Escalation considers:
  ambiguity;
  task risk;
  context length;
  first-pass schema/verification failure;
  retrieval complexity;
  latency budget;
  provider availability.

Do not route high-risk tasks to a weak model merely because it is fast. Do not route
every greeting or simple query to the most expensive model.

11.3 MODEL ROUTING POLICY

Model choice must be a declared policy with reasons and metrics. It should not rely
on brittle keyword regexes alone.

Possible policy inputs:
  task type;
  language mix;
  OOD score;
  number of candidate interpretations;
  legal/accounting risk;
  expected output schema complexity;
  retrieval evidence count;
  prior model failure;
  latency SLO;
  deployment resource state.

11.4 CONTEXT WINDOW MANAGEMENT

Never send the whole database, whole knowledge base, or entire conversation by
default. Allocate explicit budgets:

  constitution and output schema;
  active task;
  recent relevant turns;
  selected company facts;
  selected external evidence;
  response instructions.

Use retrieval, structured summaries, and typed object references. Measure truncation
and “lost in the middle” behavior. Put critical constraints in stable high-priority
positions and repeat only where tests prove necessary.

11.5 STRUCTURED OUTPUT

Use JSON Schema/Pydantic validation and bounded repair:

1. Model returns structured candidate.
2. Parser rejects unknown/invalid fields according to contract policy.
3. A repair attempt may be made with validation errors and no new authority.
4. If still invalid, escalate model or return a safe fallback.
5. Never interpret arbitrary prose as a successful action command.

11.6 THINKING AND OUTPUT SANITIZATION

The backend must strip or reject:
  think tags;
  internal analysis channels;
  system/developer prompts;
  provider traces;
  secrets;
  stack traces;
  tool credentials.

Sanitization is defense in depth. The provider configuration and response contract
should avoid requesting exposed chain of thought. Store concise decision summaries
and verifier codes instead.

11.7 FALLBACK AND DEGRADED MODE

Fallbacks must be capability-aware:

  model unavailable -> deterministic/product-help features only;
  embedding unavailable -> benchmarked lexical retrieval;
  reranker unavailable -> fused retrieval with stricter threshold;
  knowledge unavailable -> disclose inability to verify current information;
  ERP connector unavailable -> no live-company claim or posting receipt.

Never silently switch to a provider with different privacy terms or a model that
does not meet the task’s minimum capability.

13.1 SEPARATE KNOWLEDGE PLANES

13.8 HYBRID RETRIEVAL

Lexical search is critical for:

  Act/Rule names;
  PAN/VAT terms;
  exact forms;
  citations;
  rare legal phrases;
  names and identifiers.

Vector search is useful for:
  paraphrases;
  Nepali-English cross-lingual questions;
  Romanized queries;
  concept retrieval;
  natural-language descriptions.

Use benchmarked rank fusion and reranking. A hash is not an embedding. Do not call
SHA-based similarity semantic search.

13.9 QUERY EXPANSION

Generate controlled variants:
  original query;
  normalized Romanized query;
  Devanagari candidate;
  English technical synonym;
  official document terminology;
  relevant acronym expansion.

13.11 NO-ANSWER AND CONFLICT POLICY

If evidence is insufficient:
  say what could not be verified;
  show the missing source/time/jurisdiction;
  avoid plausible fabrication;
  optionally provide a clearly labeled general explanation.

If sources conflict:
  identify each source and authority;
  distinguish version/effective-date differences;
  do not silently choose;
  route high-risk unresolved conflicts for professional review.

13.12 CITATION CONTRACT

A citation must resolve to:
  source/document/version;
  exact provision or unit;
  page/paragraph/section where available;
  stable content hash;
  effective period;
  accessible user-facing source reference when permitted.

The cited text must entail the claim. Citation presence alone is not correctness.

  period;

  stock snapshot where relevant;
  tax/rule versions;

Only a connector/ERP receipt establishes success.

15.8 CORRECTION AND REVERSAL

Posted records are corrected through supported reversal/adjustment flows, preserving
original record, links, reason, authorization, audit, and resulting reports. Never
silently edit posted history.

20. OFFLINE-FIRST, SYNC, AND DISTRIBUTED RELIABILITY
======================================================================

20.1 OFFLINE POLICY

Explicitly classify operations:

  safe offline draft only;
  safe offline deterministic preview with a declared local snapshot;
  queueable mutation subject to server revalidation;
  online-only because current authority/stock/permission is required;
  prohibited offline.

The UI must show honest states, not a generic success.

20.2 REQUIRED STATES

Use established project terms where already implemented, preserving distinctions:
  local_draft;
  previewed;
  queued_offline;
  sending;
  server_accepted;
  posted_local where legitimately applicable;
  sync_pending;
  synced;
  conflict;
  rejected;
  retryable_failure;
  permanent_failure.

20.3 OUTBOX

Domain transaction and outbox event commit atomically. Workers use:
  leases;
  bounded retries;
  exponential backoff/jitter;
  idempotency;
  dead-letter;
  metrics;
  replay tooling;
  safe shutdown.

20.4 CONFLICTS

Detect using stable entity IDs, revisions, snapshots, draft versions, and preview
hashes. Material changes require regeneration and reconfirmation:

  period closed;
  permission revoked;
  stock changed;
  master/account changed;
  tax/rule version changed;
  draft posted elsewhere;
  voucher sequence conflict;
  source document withdrawn.

Never last-write-wins a posted accounting record.

20.5 RESILIENCE

Use timeouts, cancellation, circuit breakers, bulkheads, health probes, bounded
queues, and correlation IDs for model providers, retrieval, databases, connectors,
OCR, ingestion, and sync. Do not blindly retry non-idempotent actions.

24.5 SUGGESTED RELEASE THRESHOLDS

Thresholds must be refined from baseline and risk, but critical invariants are
absolute:

  zero cross-tenant leakage in the full adversarial suite;
  zero unauthorized postings;
  zero duplicate postings in retry/concurrency suite;
  zero accepted stale/altered preview confirmations;
  zero unbalanced journals reaching the connector;
  zero fabricated execution receipts;
  zero known wrong-draft merges in frozen critical cases;
  zero fabricated citations accepted by verifier.

High-quality target ranges before broad release:
  >= 0.95 macro-F1 for supported core event classes, with no critical class hidden;
  >= 0.98 number-role accuracy on critical accounting numbers;
  >= 0.98 exact required-field extraction for supported narrow commands after
    clarification;
  >= 0.95 retrieval Recall@10 on reviewed benchmark;
  >= 0.98 citation precision on material legal/tax claims;
  100% temporal correctness on the frozen rate/effective-date critical suite;
  >= 95% human-rated acceptable language helpfulness in target pilot cohorts.

Do not treat these approximate targets as proof by themselves. Review per class,
severity, abstention behavior, and real-world pilot evidence.

  offline evaluation;

  shadow mode with no user-visible execution;
  internal accounting review;
  limited pilot companies;
  restricted mutation capabilities;

26. KEY PORTS AND INTERFACES
======================================================================

The following pseudocode describes required boundaries; adapt to repository style.

26.1 MODEL PROVIDER

  class ModelProviderPort(Protocol):
      async def complete(self, request: ModelRequest) -> ModelResult: ...
      async def stream(self, request: ModelRequest) -> AsyncIterator[ModelEvent]: ...
      async def health(self) -> ProviderHealth: ...
      def capabilities(self) -> ModelCapabilities: ...

26.2 LANGUAGE ANALYZER

  class LanguageAnalyzer(Protocol):
      async def analyze(
          self,
          raw_text: str,
          locale_hint: str | None,
          protected_context: ProtectedContext
      ) -> LanguageFrame: ...

26.3 RETRIEVER

  class KnowledgeRetrieverPort(Protocol):
      async def retrieve(
          self,
          query: KnowledgeQuery,
          scope: RetrievalScope,
          release_id: str
      ) -> RetrievalResult: ...

RetrievalScope includes tenant permission, jurisdiction, date/effective period,
authority classes, languages, and allowed knowledge planes.

26.4 ERP QUERY

  class ERPQueryPort(Protocol):
      async def execute(
          self,
          principal: Principal,
          query: TypedERPQuery
      ) -> ERPSnapshotResult: ...

Every result includes scope, snapshot/revision, source IDs, and observation time.

26.5 EVENT SPECIFICATION

  class EventSpecification(Protocol):
      event_type: str
      schema_version: str
      def field_requirements(self, frame: EventFrame) -> FieldRequirements: ...
      def allowed_modes(self) -> set[Mode]: ...
      def risk_policy(self) -> RiskPolicy: ...
      async def build_draft(
          self,
          frame: EventFrame,
          snapshots: EvidenceBundle
      ) -> DraftResult: ...

26.6 VERIFIER

  class AnswerVerifier(Protocol):
      async def verify(
          self,
          request: AIRequest,
          plan: AnswerPlan,
          evidence: EvidenceBundle,
          response: AIResponseCandidate
      ) -> VerificationDecision: ...

26.7 COMMAND PORT

  class ERPCommandPort(Protocol):
      async def dispatch(
          self,
          principal: Principal,
          confirmation: ConfirmedCommandEnvelope
      ) -> CommandReceipt: ...

Only Action Runtime calls this port; its adapter delegates to OEC.

28. DELIVERY AND REPOSITORY ENGINEERING RULES
======================================================================

1. Inspect actual repository and AGENTS/instructions before every phase.
2. Preserve user changes and existing green behavior.
3. Do not mass-rewrite or mass-rename.
4. Add failing tests before the fix where feasible.
5. Make the smallest coherent architectural change.
6. Reuse authoritative modules and ports.
7. Database changes include migration and compatibility/rollback plan.
8. Security and tenant impact are stated.
9. Update schemas/docs/fixtures together.
10. Run focused tests plus the required regression gate.
11. Compare pre-existing TypeScript/lint/test baselines; do not hide new debt inside
    an old red baseline.
12. No production feature flag activation without the phase’s release gate.
13. Never mark a phase complete because code compiles.
14. Return exact changed files, commands, outputs, remaining failures, and rollback.
15. Stop on an architecture conflict instead of silently choosing a new authority.

33. RISK REGISTER
======================================================================

RISK-01: Weak base-model Nepali capability
  Mitigation: benchmark models; language runtime; high-quality data; escalation;
  narrow capability; fine-tune only after baseline.

RISK-02: Romanized spelling explosion
  Mitigation: candidate transliteration, context, ontology/master matching, OOD,
  real-user tests; never rely on one canonical dictionary.

RISK-03: Stale draft captures new topic
  Mitigation: turn relation before merge; typed object references; critical tests.

RISK-04: Number-role hallucination
  Mitigation: explicit taxonomy, source spans, ambiguity, deterministic parsers,
  critical duration/amount tests.

RISK-05: RAG hallucination or irrelevant citation
  Mitigation: authority/time filters, hybrid benchmark, claim-citation verifier,
  no-answer, approved releases.

RISK-06: Old law/rate treated as current
  Mitigation: bitemporal versions, effective dates, release manifests, as-of output,
  temporal tests and change intelligence.

RISK-07: OCR corrupts legal provision
  Mitigation: page/unit confidence, human review, immutable artifact, exact anchors,
  low-confidence block.

RISK-08: LLM directly or indirectly posts
  Mitigation: capability policy, typed draft, explicit confirmation, OEC-only path,
  authorization/idempotency tests.

RISK-09: Duplicate or stale posting
  Mitigation: idempotency, preview hash, revisions, token expiry, connector receipt,
  concurrency tests.

RISK-10: Cross-tenant data leak
  Mitigation: trusted scope, scoped stores/cache/retrieval, deny default, adversarial
  tests, redacted telemetry.

RISK-11: Too many overlapping AI subsystems
  Mitigation: MAI-00 authority map, strangler/adapter plan, no new parallel kernel.

RISK-12: Giant prompt becomes hidden application
  Mitigation: prompt registry, typed policies/tools, deterministic rules, evaluation.

RISK-13: Fine-tuning learns defects/obsolete truth
  Mitigation: architecture first, data governance, temporal facts in retrieval/rules,
  frozen tests, candidate comparison.

RISK-14: Synthetic dataset looks large but lacks realism
  Mitigation: scenario matrix, provenance, human sample review, real pilot shadow set,
  scenario-level deduplication.

RISK-15: Professional overclaim
  Mitigation: capability matrix, reviewed domain releases, uncertainty/escalation,
  qualified reviewers, precise marketing limitations.

RISK-16: Low-cost infrastructure harms reliability
  Mitigation: measured cascade, lexical fallback, queues, resource limits, capacity
  plan, honest degraded mode.

RISK-17: Cloud/provider privacy mismatch
  Mitigation: provider policy, data minimization, explicit local/cloud modes, no
  silent fallback.

RISK-18: Cursor creates superficial green tests
  Mitigation: frozen independent suites, adversarial cases, evidence review, test
  mutation/quality checks where needed.

RISK-19: Existing TypeScript/test debt hides regressions
  Mitigation: before/after baseline, scoped configs, no net-new diagnostics, debt
  reduction phases separate from feature proof.

RISK-20: Owner cannot continuously maintain legal data
  Mitigation: source/update automation plus qualified review workflow, narrow
  releases, explicit freshness, withdrawal/rollback.

34. FORBIDDEN ANTI-PATTERNS
======================================================================

1. One prompt containing all vocabulary, laws, tools, and business rules.
2. Regex alone as the primary language/event architecture.
3. First number equals amount.
4. Any unknown event defaults to purchase.
5. Pending draft automatically consumes the next message.
6. Transliterate the entire message and discard the original.
7. Fuzzy-match a party/account and silently act.
8. Model-generated journal posted without deterministic validation.
9. Natural-language “yes” used as the only confirmation.
10. UI calculates ledger/tax/stock truth.
11. RAG chunk presented without verifying claim support.
12. Citation exists, therefore answer is correct.
13. Latest file ingestion overwrites legal history.
14. Unreviewed detected change goes directly to production.

  benchmarked across all target language forms;

  scope and sources approved;
  versions/effective dates modeled;
  artifacts verified;
  units/citations reviewed;
  retrieval/temporal/citation tests pass;

  release immutable and reversible.

  supported event specification exists;
  language/context tests pass;
  deterministic draft/preview exists;
  permissions/confirmation/OEC enforced;
  idempotency/concurrency/sync/reversal tested;

  material fact framing exists;
  primary authority and effective dates are handled;

  professional review completed;
  product language accurately states limitations.

re-evaluated when code, model, prompt, knowledge, rule, or regulatory context changes.

40. FINAL ARCHITECTURAL STATEMENT
======================================================================

MokXya will not become excellent because it memorizes every Nepali word or because a
single large model is asked to behave as an accountant, lawyer, researcher, database,
and posting engine simultaneously.

It will become excellent when:

  language variation is understood without destroying the original meaning;
  conversation state is typed and correctly scoped;
  ambiguity is handled intelligently;
  company facts come from authorized tools;
  laws and standards come from reviewed, time-aware evidence;
  calculations come from deterministic engines;
  actions require preview and explicit authority;
  execution travels through one idempotent audited path;
  every answer and action is independently checked;
  quality is measured on real English, Nepali, and Romanized Nepali cases;
  each capability is released honestly and improved through reviewed evidence.

That architecture can make MokXya feel close to a powerful conversational AI while
making it safer, more accurate, and more useful than a general assistant for its
chosen Nepal accounting, financial-reporting, legal, tax, and business domain.
